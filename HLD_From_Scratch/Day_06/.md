# Introduction

Modern applications serve millions of users simultaneously.

Examples:

```text
YouTube

Amazon

Netflix

Instagram

Facebook
```

A single server cannot handle all traffic.

Without load balancing:

```text
All Requests

      |

Single Server

      |

Server Overloaded

      |

Application Crashes
```

This creates:

```text
High Latency

Server Downtime

Poor User Experience
```

Solution:

```text
Load Balancer
```

# Why Load Balancing is Needed

Imagine:

```text

Sending Requests

```

# What is a Load Balancer?

A load balancer is a server that distributes incoming network traffic across multiple backend servers.

Goal:

```text
Distribute Traffic Evenly

Prevent Server Overload

Ensure High Availability

Improve Scalability
```

Think of load balancer as:

```text
A Traffic Police

Directing Vehicles

To Different Roads
```

# Round Robin

Requests distributed to each server in order.

```text
Request 1  →  Server 1

Request 2  →  Server 2

Request 3  →  Server 3

Request 4  →  Server 1

Request 5  →  Server 2

Request 6  →  Server 3
```

Simple rotation.

---

## ASCII Diagram

```text
          Load Balancer

               |

    +----------+----------+

    |                     |

Request 1            Request 2

    |                     |

 Server 1             Server 2

Request 3

    |

 Server 3

Request 4

    |

 Server 1  (cycle repeats)
```

---

## When to Use

```text
All Servers Have Equal Capacity

Simple Setup Required

Stateless Applications
```

---

## Limitation

```text
Does Not Consider Server Load

Heavy Request On Server 1

Light Request On Server 2

Still Sends To Server 1 In Turn
```

# Weighted Round Robin

Each server assigned a weight based on capacity.

```text
Server 1  →  Weight 3  (powerful)

Server 2  →  Weight 2  (medium)

Server 3  →  Weight 1  (weak)
```

Distribution:

```text
Request 1  →  Server 1

Request 2  →  Server 1

Request 3  →  Server 1

Request 4  →  Server 2

Request 5  →  Server 2

Request 6  →  Server 3
```

---

## ASCII Diagram

```text
          Load Balancer

               |

    +----------+----------+

    |          |          |

 Server 1   Server 2   Server 3

 Weight 3   Weight 2   Weight 1

 3 reqs     2 reqs     1 req

 per cycle  per cycle  per cycle
```

---

## When to Use

```text
Servers Have Different Capacities

New Server Added With Less Power

Gradual Traffic Migration
```

# Least Connections

Request sent to server with fewest active connections.

```text
Server 1  →  10 Active Connections

Server 2  →  3 Active Connections

Server 3  →  7 Active Connections

New Request  →  Server 2
```

---


---

## When to Use

```text
Requests Have Variable Processing Time

Long Running Connections

WebSocket Applications
```

---

# Weighted Least Connections

Combines weight and active connections.

Formula:

```text
Score = Active Connections / Weight

Lowest Score Gets Next Request
```

Example:

```text
Server 1  →  Connections 6  Weight 3  Score 2.0

Server 2  →  Connections 4  Weight 2  Score 2.0

Server 3  →  Connections 1  Weight 1  Score 1.0

New Request  →  Server 3
```

---

## When to Use

```text
Servers Have Different Capacities

And Variable Request Processing Times
```

# IP Hash

Client IP address determines which server receives request.

```text
Hash(Client IP)  →  Server Index
```

Example:

```text
IP 192.168.1.1  →  Hash → Server 1

IP 192.168.1.2  →  Hash → Server 2

IP 192.168.1.1  →  Hash → Server 1  (always same)
```

Same client always goes to same server.

## When to Use

```text
Session Data Stored On Server
```

# Resource Based

Load balancer checks actual server resources before routing.

```text
Server 1  →  CPU 90%   Memory 80%

Server 2  →  CPU 20%   Memory 30%

Server 3  →  CPU 50%   Memory 60%

New Request  →  Server 2
```

Most intelligent algorithm.

Requires monitoring agent on each server.

# Layer 4 vs Layer 7 Load Balancing

```text
OSI Model

      |

      +---- Layer 4  (Transport Layer)

      |

      +---- Layer 7  (Application Layer)
```

---

## Layer 4 Load Balancing

Works on TCP/UDP level.

```text
Sees:

IP Address

Port Number
```

Does not read request content.

Faster but less intelligent.

---

## Layer 7 Load Balancing

Works on HTTP level.

```text
Sees:

URL Path

Headers

Cookies

Request Body
```

Example:

```text
/api/users    →  User Service

/api/products →  Product Service

/api/orders   →  Order Service
```

Slower but more intelligent.

---

## Comparison

```text
Feature         Layer 4        Layer 7

Speed           Faster         Slower

Intelligence    Low            High

Content Aware   No             Yes

Use Case        TCP Apps       HTTP Apps
```

# Health Checks

Load balancer continuously monitors server health.

```text
Every 5 Seconds

Load Balancer Pings Each Server

      |

 +----+----+

 |         |

Healthy   Unhealthy

 |         |

Keep      Remove From Pool
```

Types:

```text
TCP Health Check

HTTP Health Check

Custom Script Check
```

---

## Example

```text
Server 1  →  Responds 200 OK   →  Healthy

Server 2  →  No Response       →  Unhealthy  →  Removed

Server 3  →  Responds 200 OK   →  Healthy
```

Traffic automatically rerouted away from failed server.

# Interview Questions

### What is a load balancer?

A load balancer distributes incoming traffic across multiple servers to prevent overload and ensure availability.

---

### What is Round Robin?

Requests distributed to servers in sequential order regardless of server load.

---

### Difference between Layer 4 and Layer 7?

```text
Layer 4  →  TCP level, IP and port only

Layer 7  →  HTTP level, reads URL, headers, cookies
```

---

### What is IP Hash?

Client IP is hashed to determine which server handles all requests from that client.

---

### What are sticky sessions?

Mechanism to ensure same client always reaches same server, used when session data is stored locally on server.

---

### What is a health check?

Periodic ping from load balancer to servers to detect failures and remove unhealthy servers from the pool.

---

### When to use Least Connections?

When requests have variable processing time such as video processing, file uploads, or WebSocket connections.

---

# Summary

- Load balancers distribute traffic across multiple servers.
- Prevents any single server from becoming a bottleneck.
- Round Robin is simplest, suitable for equal capacity servers.
- Weighted Round Robin handles servers of different capacities.
- Least Connections is best for variable length requests.
- IP Hash ensures same client reaches same server always.
- Layer 4 is faster, Layer 7 is smarter.
- Health checks automatically remove failed servers from pool.
- Sticky sessions maintain user session on same server.
- Netflix, Amazon, and Cloudflare rely heavily on load balancing.
