## Introduction

Multiple transactions run at the same time on a database.

```
Transaction 1
Transaction 2
Transaction 3
```

Without control, transactions interfere with each other.

```
Two Transactions
      |
Read/Write Same Row
      |
Inconsistent Data
```

**Solution:**

```
Isolation Levels
```

---

## Why Isolation is Needed

```
User A Transfers Money
User B Reads Balance
Same Time
```

Without isolation, User B may see incorrect or half-updated data.

---

## What is Isolation?

Isolation defines how much one transaction is visible to another before it commits.

```
Higher Isolation → More Consistency, Less Concurrency
Lower Isolation  → More Concurrency, Less Consistency
```

---

## Read Phenomena

```
Problems
    |
    +---- Dirty Read
    |
    +---- Non-Repeatable Read
    |
    +---- Phantom Read
```

### Dirty Read

Reading uncommitted data of another transaction.

```
T1: Update balance = 500 (not committed)
T2: Reads balance = 500
T1: Rollback
T2: Has wrong data
```

### Non-Repeatable Read

Same row gives different values within one transaction.

```
T1: Read balance = 500
T2: Update balance = 700, Commit
T1: Read again → balance = 700
```

### Phantom Read

New rows appear in a repeated query.

```
T1: SELECT * WHERE age > 20 → 5 rows
T2: Insert new row, Commit
T1: Same query again → 6 rows
```

---

## Isolation Levels

```
Read Uncommitted
Read Committed
Repeatable Read
Serializable
```

### Read Uncommitted

```
Dirty Read     → Possible
Non-Repeatable → Possible
Phantom Read   → Possible
```

Lowest isolation, highest performance.

### Read Committed

```
Dirty Read     → Prevented
Non-Repeatable → Possible
Phantom Read   → Possible
```

Default in PostgreSQL, Oracle, SQL Server.

### Repeatable Read

```
Dirty Read     → Prevented
Non-Repeatable → Prevented
Phantom Read   → Possible
```

Default in MySQL InnoDB.

### Serializable

```
Dirty Read     → Prevented
Non-Repeatable → Prevented
Phantom Read   → Prevented
```

Transactions behave as if executed one by one.
Highest consistency, lowest concurrency.

---

## Isolation vs Performance Tradeoff

```
Read Uncommitted  →  Fastest,  Least Safe
Read Committed     →  Balanced
Repeatable Read     →  Safer
Serializable        →  Slowest, Safest
```

---

## Real-World Examples

```
Banking Systems       → Serializable
E-Commerce Checkout   → Repeatable Read
Analytics Dashboards  → Read Committed
```

---

## Interview Questions

**What is a dirty read?**
Reading data that another transaction has written but not yet committed.

**Default isolation level in PostgreSQL?**
Read Committed.

**Which level prevents phantom reads?**
Serializable.

**Difference between Repeatable Read and Serializable?**
Repeatable Read locks read rows but new rows can still appear (phantom); Serializable prevents that entirely by treating transactions as fully sequential.

---

## Summary

- Isolation controls transaction visibility to each other.
- Four standard levels: Read Uncommitted, Read Committed, Repeatable Read, Serializable.
- Higher isolation = more consistency, less concurrency.
- Databases pick different defaults based on use case.
