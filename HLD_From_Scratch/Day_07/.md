# Introduction

Modern distributed systems run across multiple servers.

Examples:

```text
PostgreSQL Clusters

MongoDB Replica Sets

MySQL Master Slave Setup
```

A single database server is a single point of failure.

Without replication:

```text
Database Server Crashes

      |

All Data Inaccessible

      |

Application Down

      |

Users Cannot Access Anything
```

This creates:

```text
Data Loss

Downtime

Revenue Loss
```

Solution:

```text
Data Replication

Circuit Breaker

Data Recovery
```

# What is Data Replication?

Replication is the process of copying and maintaining database copies across multiple servers.

Goal:

```text
High Availability

Fault Tolerance

Read Scalability

Disaster Recovery
```

Think of replication as:

```text
A Backup That Is Always Live

And Always In Sync
```

---

# How Replication Works

```text
Client Write Request

        |

        v

  Primary Database

        |

        v

  Write to Own DB

        |

        v

  Log the Write (WAL)

        |

        v

  Send to Replicas

        |

   +----+----+

   |         |

Replica 1  Replica 2

   |         |

Apply      Apply

Write      Write
```

Reads can be served from replicas.

Writes always go to primary.

```text
Replication

      |

      +---- Single Leader

```

# Single Leader Replication

One node accepts all writes.

All other nodes are read-only replicas.

```text
         Primary

        (Writes)

            |

    +-------+-------+

    |               |

 Replica 1       Replica 2

  (Reads)          (Reads)
```

Most common pattern.

Used by:

```text
PostgreSQL

MySQL

MongoDB
```

---

## Advantages

```text
Simple to Implement

No Write Conflicts

Easy to Reason About
```

---

## Disadvantages

```text
Primary is Single Point of Failure

Write Bottleneck On Primary

Replication Lag Possible
```

---

# Multi Leader Replication

Multiple nodes accept writes.

```text
   Leader 1        Leader 2

   (Writes)        (Writes)

       |               |

       +-------+-------+

               |

           Replicate

           Each Other
```

Used across multiple data centers.

```text
Data Center India     Data Center USA

    Leader 1              Leader 2

       |                     |

   Replicas              Replicas
```

---

## Advantages

```text
Write Availability In Each Data Center

Lower Latency For Writes

Survives Data Center Failure
```

---

## Disadvantages

```text
Write Conflicts Possible

Complex Conflict Resolution

Harder to Implement
```

---

# Leaderless Replication

Any node can accept writes.

Client writes to multiple nodes simultaneously.

```text
Client Write

      |

+-----+-----+

|     |     |

N1    N2    N3

All Accept Write
```

Used by:

```text
Apache Cassandra

Amazon DynamoDB
```

# Synchronous vs Asynchronous Replication

```text
Replication Timing

      |

      +---- Synchronous

      |

      +---- Asynchronous
```

---

## Synchronous Replication

Primary waits for replica to confirm before responding to client.

```text
Client Write

      |

   Primary

      |

   Replica   (must confirm)

      |

   Primary Responds to Client
```

Advantages:

```text
Zero Data Loss

Replica Always Up To Date
```

Disadvantages:

```text
Slower Response Time

Primary Blocked Until Replica Confirms
```

---

## Asynchronous Replication

Primary responds to client immediately.

Replication happens in background.

```text
Client Write

      |

   Primary

      |

Client Gets Response  (immediately)

      |

   Background

      |

   Replica Updated Later
```

Advantages:

```text
Faster Response Time

Primary Not Blocked
```

Disadvantages:

```text
Replica May Lag Behind

Data Loss If Primary Crashes Before Replication
```

---

# Replication Lag

Delay between primary write and replica update.

```text
Primary  →  Write at 10:00:00.000

Replica  →  Updated at 10:00:00.150

Lag      →  150 milliseconds
```

During lag:

```text
User Writes Data

      |

User Reads Data From Replica

      |

Sees Old Data
```

This is called:

```text
Read Your Own Write Problem
```

---

# Write-Ahead Log

WAL is the foundation of replication and recovery.

Every write is logged before being applied.

```text
Write Request Arrives

        |

        v

Log Entry Written to WAL

        |

        v

Write Applied to Database

        |

        v

WAL Entry Sent to Replicas
```

WAL contains:

```text
Operation Type   (INSERT, UPDATE, DELETE)

Document ID

Full Data Snapshot

Timestamp
```

If replica crashes:

```text
Replica Restarts

      |

Reads WAL From Primary

      |

Replays Missing Entries

      |

Back In Sync
```

# What is a Circuit Breaker?

Circuit breaker is a design pattern that prevents cascading failures.

Named after electrical circuit breakers.

```text
Electrical Circuit Breaker

      |

Too Much Current

      |

Breaker Trips

      |

Circuit Opens

      |

No More Current Flows

      |

Prevents Fire
```

In software:

```text
Service Failing Repeatedly

      |

Circuit Breaker Trips

      |

Requests Blocked

      |

No More Calls To Failed Service

      |

Prevents System Overload
```

---

# Why Circuit Breaker is Needed

Without circuit breaker:

```text
Replica Down

      |

Every Request Tries To Hit Replica

      |

Every Request Waits For Timeout

      |

Threads Pile Up

      |

Entire Application Slows Down
```

With circuit breaker:

```text
Replica Down

      |

3 Failures Detected

      |

Circuit Opens

      |

Requests Blocked Immediately

      |

Fallback To Primary

      |

No Thread Pileup
```

---

# Circuit Breaker States

```text
States

   |

   +---- CLOSED

   |

   +---- OPEN

   |

   +---- HALF_OPEN
```

---

## CLOSED State

Everything is working normally.

```text
Request

   |

Circuit Breaker (CLOSED)

   |

Service

   |

Response
```

All requests pass through.

---

## OPEN State

Too many failures detected.

```text
Request

   |

Circuit Breaker (OPEN)

   |

Blocked Immediately

   |

Fallback Response
```

No requests reach the failed service.

---

## HALF_OPEN State

Recovery timeout has passed.

One test request allowed through.

```text
Request

   |

Circuit Breaker (HALF_OPEN)

   |

One Test Request

   |

 +------+------+

 |             |

Success      Failure

 |             |

CLOSED        OPEN Again
```

---

# Circuit Breaker Flow

```text
Start → CLOSED

Failure 1  →  failureCount = 1

Failure 2  →  failureCount = 2

Failure 3  →  failureCount = 3  →  Circuit OPENS

Next Request  →  Blocked Immediately

15 Seconds Pass  →  HALF_OPEN

Test Request Sent

      |

 +----+----+

 |         |

Pass      Fail

 |         |

CLOSED   OPEN Again
```

# What is Data Recovery?

Data recovery is the process of restoring a replica to a consistent state after a crash.

Without recovery:

```text
Replica Crashes

      |

Replica Restarts

      |

Has Old Data

      |

Out Of Sync With Primary

      |

Serves Stale Data To Users
```

With WAL based recovery:

```text
Replica Crashes

      |

Replica Restarts

      |

Reads Pending WAL Entries From Primary

      |

Replays Each Entry In Order

      |

Back In Sync

      |

Serves Fresh Data
```

---

# How Recovery Works

```text
Step 1

Replica comes back online

Step 2

Recovery process reads WAL

Find all entries where appliedToReplica = false

Step 3

For each pending entry

Apply operation to replica database

Step 4

Mark WAL entry as appliedToReplica = true

Step 5

Replica is now fully in sync
```

---

# WAL Replay

Entries replayed in chronological order.

```text
WAL Entry 1  →  INSERT  Product A  →  Applied

WAL Entry 2  →  UPDATE  Product A  →  Applied

WAL Entry 3  →  INSERT  Product B  →  Applied

WAL Entry 4  →  DELETE  Product C  →  Applied

All Entries Applied

Replica In Sync
```

Order matters.

Replaying out of order causes inconsistency.

---

# Failover

When primary crashes, a replica is promoted to primary.

```text
Primary Crashes

      |

Replica 1 Detected No Heartbeat

      |

Leader Election

      |

Replica 1 Promoted To Primary

      |

Other Replicas Point To New Primary

      |

Application Continues
```

Types:

```text
Manual Failover   →  Human triggers promotion

Automatic Failover  →  System detects and promotes automatically
```

# Interview Questions

### What is data replication?

Copying and maintaining identical data across multiple database servers for availability and fault tolerance.

---

### What is the difference between synchronous and asynchronous replication?

```text
Synchronous   →  Primary waits for replica confirmation before responding

Asynchronous  →  Primary responds immediately, replica updated in background
```

---

### What is a Write-Ahead Log?

A log where every write is recorded before being applied to the database. Used for replication and recovery.

---

### What is replication lag?

The delay between a write being applied on the primary and the same write appearing on the replica.

---

### What are the three states of a circuit breaker?

```text
CLOSED     →  Normal operation, requests pass through

OPEN       →  Failures detected, requests blocked

HALF_OPEN  →  Testing if service recovered
```

---

### What is data recovery in the context of replication?

Replaying pending WAL entries on a replica that was down, to bring it back in sync with the primary.

---

### What is failover?

The process of promoting a replica to primary when the original primary crashes.

---

### What is the difference between failover and recovery?

```text
Recovery  →  Replica catches up using WAL after coming back online

Failover  →  Replica is promoted to primary when primary crashes
```

---

# Summary

- Replication maintains copies of data across multiple servers.
- Single leader replication is the most common pattern.
- Multi leader replication enables multi data center writes.
- WAL logs every write for replication and recovery purposes.
- Synchronous replication is safer but slower.
- Asynchronous replication is faster but risks data loss.
- Circuit breaker has three states: CLOSED, OPEN, and HALF_OPEN.
- Circuit breaker prevents cascading failures when a service is down.
- Data recovery replays WAL entries to bring replica back in sync.
- PostgreSQL, MySQL, and MongoDB all use WAL based replication.
