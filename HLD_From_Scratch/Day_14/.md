## Introduction

```
Distributing Data Across Servers
```

Server count changes often (scale up/down, crashes).

---

## Why Consistent Hashing is Needed

```
Servers: 3
Key % 3 → Server Index
```

If a server is added or removed, almost all keys remap to different servers.

---

## Problem with Normal Hashing

```
key % N (N = number of servers)

N = 3 → N = 4
      |
Almost All Keys Remapped
      |
Massive Cache Misses
```

---

## What is Consistent Hashing?

A hashing technique where only a small fraction of keys need remapping when servers are added or removed.

```
Hash Space Arranged as a Circle (Ring)
```

---

## Hash Ring

```
        Server A
       /        \
  Server D      Server B
       \        /
        Server C
```

Both servers and keys are hashed onto the same ring.

```
Key is Assigned to
First Server Found
Moving Clockwise
```

---

## Adding a Node

```
New Server E Added to Ring
      |
Only Keys Between E and
Its Previous Neighbor Move
      |
Rest of the Keys Stay Same
```

---

## Removing a Node

```
Server B Crashes
      |
Its Keys Move to Next Server
Clockwise (Server C)
      |
Other Servers Unaffected
```

---

## Virtual Nodes

```
Problem: Uneven Load Distribution
```

```
Each Physical Server
      |
Mapped to Multiple Points on Ring
      |
Better Load Balance
```

```
Server A → A1, A2, A3 (virtual nodes)
Server B → B1, B2, B3
```

---

## Real-World Examples

```
DynamoDB    → Uses consistent hashing for partitioning
Cassandra   → Ring-based data distribution
Memcached   → Client-side consistent hashing
CDNs        → Route requests to nearest cache node
```

---

## Interview Questions

**Why is normal modulo hashing bad for distributed systems?**
Because adding or removing a server changes the modulo base, causing almost all keys to remap and creating massive cache misses.

**What problem do virtual nodes solve?**
They solve uneven load distribution by spreading each physical server across multiple points on the ring instead of just one.

**How many keys move when a server is added in consistent hashing?**
Only the keys between the new server and its nearest neighbor on the ring move; the rest stay in place.

---

## Summary

- Consistent hashing minimizes remapping when servers change.
- Keys and servers both hashed onto the same ring.
- Only a small fraction of keys move on add/remove.
- Virtual nodes fix uneven load distribution.
- Used by DynamoDB, Cassandra, Memcached, CDNs.
