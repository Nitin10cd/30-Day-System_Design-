# Dependency Inversion Principle (DIP)
## SOLID Principles — The "D"

---

## Definition

> **"High-level modules should not depend on low-level modules. Both should depend on abstractions."**
>
> **"Abstractions should not depend on details. Details should depend on abstractions."**
>
> — Robert C. Martin (Uncle Bob)

The Dependency Inversion Principle flips the traditional dependency direction. Instead of high-level business logic directly using low-level implementations, **both layers depend on an abstract interface in the middle**. This is called "inverting" the dependency.

---

## Understanding the Two Parts

### Part 1: High-level modules should NOT depend on low-level modules

| Term | Meaning | Example |
|------|---------|---------|
| **High-level module** | Business logic, policies, orchestration | `OrderProcessor` |
| **Low-level module** | Infrastructure, I/O, concrete tools | `MySQLDatabase`, `EmailSender` |
| **Abstraction** | Interface/abstract class that both depend on | `IDatabase`, `INotifier` |

Without DIP: `OrderProcessor → MySQLDatabase` (direct dependency, tightly coupled)

With DIP: `OrderProcessor → IDatabase ← MySQLDatabase` (both depend on abstraction)

### Part 2: Abstractions should not depend on details

The interface `IDatabase` should NOT include MySQL-specific methods like `runMySQLQuery()`. The interface is general; the concrete class is the detail.

---

## The Problem: Direct Dependencies

### Without DIP (VIOLATION)

```cpp
// Low-level module — concrete implementation
class MySQLDatabase {
public:
    void saveOrder(const Order& order) {
        // MySQL-specific code
        std::cout << "Saving to MySQL: " << order.id << std::endl;
    }
};

class EmailSender {
public:
    void sendConfirmation(const std::string& email) {
        // SMTP-specific code
        std::cout << "Sending email to: " << email << std::endl;
    }
};

// High-level module — DIRECTLY depends on low-level modules
class OrderProcessor {
private:
    MySQLDatabase db;      // ❌ Hardcoded to MySQL
    EmailSender emailer;   // ❌ Hardcoded to Email

public:
    void processOrder(const Order& order) {
        db.saveOrder(order);
        emailer.sendConfirmation(order.customerEmail);
    }
};
```

**Problems:**
- `OrderProcessor` is hardcoded to MySQL — switching to PostgreSQL requires modifying `OrderProcessor`
- `OrderProcessor` is hardcoded to Email — switching to SMS requires modifying `OrderProcessor`
- Cannot unit test `OrderProcessor` without a real MySQL database
- High-level business logic is polluted by infrastructure concerns
- Every new technology choice forces changes in business logic

---

## The Solution: Depend on Abstractions

### With DIP (CORRECT)

```cpp
// Abstractions (interfaces)
class IDatabase {
public:
    virtual void saveOrder(const Order& order) = 0;
    virtual ~IDatabase() = default;
};

class INotifier {
public:
    virtual void sendConfirmation(const std::string& contact) = 0;
    virtual ~INotifier() = default;
};

// Low-level modules implement the abstractions
class MySQLDatabase : public IDatabase {
public:
    void saveOrder(const Order& order) override {
        std::cout << "MySQL: Saving order " << order.id << std::endl;
    }
};

class PostgreSQLDatabase : public IDatabase {
public:
    void saveOrder(const Order& order) override {
        std::cout << "PostgreSQL: Saving order " << order.id << std::endl;
    }
};

class EmailNotifier : public INotifier {
public:
    void sendConfirmation(const std::string& email) override {
        std::cout << "Email: Confirmation sent to " << email << std::endl;
    }
};

class SMSNotifier : public INotifier {
public:
    void sendConfirmation(const std::string& phone) override {
        std::cout << "SMS: Confirmation sent to " << phone << std::endl;
    }
};

// High-level module — depends only on abstractions
class OrderProcessor {
private:
    IDatabase& db;         // ✅ Depends on abstraction
    INotifier& notifier;   // ✅ Depends on abstraction

public:
    // Dependencies injected from outside (Dependency Injection)
    OrderProcessor(IDatabase& db, INotifier& notifier)
        : db(db), notifier(notifier) {}

    void processOrder(const Order& order) {
        db.saveOrder(order);
        notifier.sendConfirmation(order.customerEmail);
    }
};

// Usage — wiring happens at the top (composition root)
int main() {
    MySQLDatabase mysql;
    EmailNotifier emailer;
    OrderProcessor processor(mysql, emailer);
    processor.processOrder(someOrder);

    // Switching is trivial — no changes to OrderProcessor
    PostgreSQLDatabase postgres;
    SMSNotifier sms;
    OrderProcessor processor2(postgres, sms);
}
```

## Dependency Injection (DI): How DIP is Implemented

DIP says **what** to depend on (abstractions). Dependency Injection says **how** to provide those dependencies. There are three common patterns:

### 1. Constructor Injection (Preferred)

```cpp
class OrderProcessor {
public:
    OrderProcessor(IDatabase& db, INotifier& notifier)
        : db(db), notifier(notifier) {}
private:
    IDatabase& db;
    INotifier& notifier;
};
```
✅ Dependencies are explicit, required, and immutable after construction.

### 2. Setter Injection

```cpp
class OrderProcessor {
public:
    void setDatabase(IDatabase* db) { this->db = db; }
    void setNotifier(INotifier* notifier) { this->notifier = notifier; }
private:
    IDatabase* db = nullptr;
    INotifier* notifier = nullptr;
};
```
⚠️ Allows optional or swappable dependencies, but object can be used before being configured.

### 3. Interface Injection

```cpp
class IInjectDatabase {
public:
    virtual void injectDatabase(IDatabase& db) = 0;
};

class OrderProcessor : public IInjectDatabase {
public:
    void injectDatabase(IDatabase& db) override { this->db = &db; }
};
```
⚠️ Less common in C++, more common in Java/C# frameworks.

## Core Concepts

### The Traditional Dependency Direction (Before DIP)

```
[UI Layer]
    ↓
[Business Logic Layer]        ← high-level
    ↓
[Data Access Layer]           ← low-level
    ↓
[Database]
```

Each layer directly depends on the layer below it. Changes in `Database` force changes up the chain.

### The Inverted Dependency Direction (After DIP)

```
[UI Layer]
    ↓
[Business Logic Layer] ──→ [IDatabase Interface]
                                    ↑
                          [Data Access Layer]
                                    ↑
                             [Database]
```

Now `Business Logic` owns the interface. `Data Access` must conform to what `Business Logic` needs — not the other way around. The dependency is inverted.

## Real-World Analogies

### Analogy 1: Power Outlets
- A laptop doesn't depend on a **specific power plant**
- It depends on the **power outlet standard** (120V/240V, plug shape)
- The power plant must conform to the standard — not the laptop
- ISP analogy: the outlet is the abstraction; the power plant is the low-level detail

### Analogy 2: Light Switch
- A light switch doesn't care if it controls an **LED bulb, fluorescent, or smart bulb**
- It uses an abstraction: "something that turns on/off"
- The switch (high-level) and bulb (low-level) both depend on this abstract contract

### Analogy 3: USB Standard
- Your computer depends on the **USB protocol**, not on a specific mouse brand
- Any USB-compliant mouse works
- The abstraction (USB) is what both depend on

## The Composition Root Pattern

When using DIP, all the "wiring" (connecting abstractions to implementations) should happen in one place: the **composition root**. This is usually `main()` or an application startup class.

```cpp
int main() {
    // Composition root — the ONLY place that knows about concrete types
    MySQLDatabase db;
    EmailNotifier notifier;
    Logger logger;

    OrderProcessor processor(db, notifier);
    ReportGenerator reporter(db, logger);
    // etc.
}
```

Everything else in the application only knows about interfaces.

## Testing Benefits

DIP makes unit testing clean and straightforward:

```cpp
// Mock implementations for testing
class MockDatabase : public IDatabase {
public:
    std::vector<Order> savedOrders;
    void saveOrder(const Order& order) override {
        savedOrders.push_back(order);  // Just record it
    }
};

class MockNotifier : public INotifier {
public:
    std::vector<std::string> sentMessages;
    void sendConfirmation(const std::string& contact) override {
        sentMessages.push_back(contact);
    }
};

// Test without any real database or email server!
void testOrderProcessor() {
    MockDatabase mockDB;
    MockNotifier mockNotifier;
    OrderProcessor processor(mockDB, mockNotifier);

    Order testOrder{1, "test@example.com"};
    processor.processOrder(testOrder);

    assert(mockDB.savedOrders.size() == 1);
    assert(mockNotifier.sentMessages[0] == "test@example.com");
    std::cout << "All tests passed!" << std::endl;
}
```

## Signs You're Violating DIP

| Smell | What It Means |
|-------|---------------|
| `new ConcreteClass()` inside a high-level class | High-level is creating its own dependencies |
| `#include "MySQLDatabase.h"` in business logic | Direct coupling to concrete module |
| "I can't test this without a real database" | Missing abstraction layer |
| Changing a database requires editing business logic | Tight coupling |
| Conditional `if (useMySQL) ... else if (usePostgres)` in business logic | Logic that should be in concrete classes leaking up |

---

## DIP vs Dependency Injection

These are often confused. They are related but different:

| Concept | What It Is | Level |
|---------|-----------|-------|
| **DIP** | Design principle — depend on abstractions | Architecture |
| **DI** | Pattern — inject dependencies from outside | Implementation |
| **IoC Container** | Framework that automates DI | Framework |

DIP is the **goal**. DI is one **technique** to achieve it.

---

## Benefits of Following DIP

**1. Loose Coupling**
High-level modules are insulated from changes in low-level modules.

**2. Testability**
Dependencies can be replaced with mocks/stubs in tests without touching production code.

**3. Flexibility and Interchangeability**
Swap implementations (MySQL → PostgreSQL, Email → SMS) by changing the composition root only.

**4. Parallel Development**
Teams can work on high-level and low-level modules simultaneously, agreeing on interfaces first.

**5. Reusability**
High-level modules that depend only on abstractions can be reused across different projects with different implementations.

---

## Summary

| Aspect | Details |
|--------|---------|
| **Letter** | D (5th and final principle) |
| **Author** | Robert C. Martin |
| **Core idea** | High-level and low-level modules both depend on abstractions |
| **Mechanism** | Pure abstract classes / interfaces as the middle layer |
| **Implementation** | Dependency Injection (constructor, setter, or interface) |
| **Key question** | "Does my business logic reference any concrete class?" |
| **Red flag** | `new ConcreteClass()` inside a class you want to be testable |
| **Goal** | Decouple what you do from how you do it |

---

> DIP is perhaps the most powerful of the SOLID principles. It creates a clear separation between **policy** (what the system does) and **mechanism** (how it does it), making systems dramatically easier to change, test, and extend.
