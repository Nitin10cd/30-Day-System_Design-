# Interface Segregation Principle (ISP)
## SOLID Principles — The "I"

---

## Definition

> **"Clients should not be forced to depend on interfaces they do not use."**
>
> — Robert C. Martin (Uncle Bob)

The Interface Segregation Principle states that a class should **never be forced to implement methods it doesn't need**. Instead of one large, fat interface, you should have many small, focused, role-specific interfaces. Each interface should serve **one client type** and contain **only what that client actually uses**.

---

## The Problem: Fat Interfaces

A **fat interface** (also called a "polluted" or "bloated" interface) is one that forces implementing classes to provide methods that make no sense for them.

### Example of a Fat Interface (VIOLATION)

```cpp
// BAD: One giant interface for all machine types
class IMachine {
public:
    virtual void print(Document& doc) = 0;
    virtual void scan(Document& doc) = 0;
    virtual void fax(Document& doc) = 0;
    virtual void staple(Document& doc) = 0;
    virtual void copy(Document& doc) = 0;
};

// A simple printer is FORCED to implement fax, scan, staple, copy
class OldPrinter : public IMachine {
public:
    void print(Document& doc) override { /* actual printing */ }
    void scan(Document& doc) override { throw std::runtime_error("Not supported!"); }
    void fax(Document& doc) override { throw std::runtime_error("Not supported!"); }
    void staple(Document& doc) override { throw std::runtime_error("Not supported!"); }
    void copy(Document& doc) override { throw std::runtime_error("Not supported!"); }
};
```

**Why this is BAD:**
- `OldPrinter` is forced to implement `scan`, `fax`, `staple`, `copy` — which it cannot do
- Clients using `IMachine` might accidentally call `scan()` on an `OldPrinter` and get a runtime crash
- Every time you add a method to `IMachine`, ALL classes must change
- Violates the principle of least surprise

---

## The Solution: Segregated Interfaces

Split the fat interface into small, focused role interfaces.

### Example Following ISP (CORRECT)

```cpp
// GOOD: Small, focused interfaces
class IPrinter {
public:
    virtual void print(Document& doc) = 0;
    virtual ~IPrinter() = default;
};

class IScanner {
public:
    virtual void scan(Document& doc) = 0;
    virtual ~IScanner() = default;
};

class IFax {
public:
    virtual void fax(Document& doc) = 0;
    virtual ~IFax() = default;
};

// A simple printer only implements what it CAN do
class OldPrinter : public IPrinter {
public:
    void print(Document& doc) override { /* actual printing */ }
};

// A modern multifunctional device implements multiple interfaces
class ModernPrinter : public IPrinter, public IScanner, public IFax {
public:
    void print(Document& doc) override { /* print */ }
    void scan(Document& doc) override { /* scan */ }
    void fax(Document& doc) override { /* fax */ }
};
```

**Why this is GOOD:**
- `OldPrinter` only implements what it can actually do
- Clients depend only on the interface they need
- Adding a new interface doesn't break existing classes
- Code is easier to read, test, and maintain

## Core Concepts

### 1. Role Interfaces vs Header Interfaces

| Type | Description | Example |
|------|-------------|---------|
| **Header Interface** | One interface listing ALL methods of a class | `IMachine` with 10 methods |
| **Role Interface** | Small interface for a specific client role | `IPrinter`, `IScanner`, `IFax` |

ISP favors **role interfaces** — define interfaces from the **client's perspective**, not the implementation's.

### 2. Interface Cohesion

An interface is cohesive when all its methods are used together by the same client. If you find yourself writing `throw NotImplemented()` in multiple places, the interface is not cohesive.

### 3. Composition over Pollution

When a class needs to do many things, it can implement **multiple small interfaces** rather than one large one:

```cpp
class AllInOnePrinter : public IPrinter,
                        public IScanner,
                        public IFax,
                        public ICopier {
    // Implements all capabilities cleanly
};
```

## Real-World Analogies

### Analogy 1: TV Remote vs Universal Remote
- A **basic TV remote** has: power, volume, channel — that's it
- A **universal remote** has 80 buttons — confusing, hard to use
- ISP says: give clients **only the controls they need**

### Analogy 2: Restaurant Menu
- A children's menu has 5 items — simple, relevant
- A full menu has 100 items — overwhelming for a child
- ISP says: design the interface for **who will use it**

### Analogy 3: Job Descriptions
- An interface is like a **job description**
- A `Surgeon` interface shouldn't require `performAccounting()` just because the hospital has an all-purpose `IEmployee` interface
- Split responsibilities into focused roles

## Signs You're Violating ISP

| Smell | What It Means |
|-------|---------------|
| `throw NotImplementedException()` | Method forced by interface but not needed |
| Empty method bodies with no logic | Same as above |
| One interface imported for one method | Interface is too large for this client |
| Recompiling unrelated classes when interface changes | Interface serves too many masters |
| "I only use 2 of these 15 methods" | Interface needs segregation |

---

## Benefits of Following ISP

**1. Reduced Coupling**
Classes only depend on what they actually use. Changes to one interface don't ripple through unrelated classes.

**2. Easier Testing**
Small interfaces are easy to mock. You can test `IPrinter` behavior independently of `IScanner`.

**3. Higher Cohesion**
Each interface has a single, clear purpose. This makes the codebase self-documenting.

**4. Flexible Composition**
New device types can pick and choose exactly which interfaces to implement without being burdened by the rest.

**5. Follows Open/Closed Principle**
When you add a new capability (like `IStapler`), existing classes are untouched.

## ISP and Other SOLID Principles

| Principle | Relationship to ISP |
|-----------|---------------------|
| **SRP** | Both promote single-purpose design — SRP for classes, ISP for interfaces |
| **OCP** | ISP enables OCP: adding new interfaces doesn't break existing code |
| **LSP** | ISP prevents LSP violations caused by empty/throwing implementations |
| **DIP** | ISP helps DIP: small focused interfaces are better abstractions to depend on |

## Common Misconceptions

**"ISP means every interface should have exactly one method"**
No. It means every method in an interface should be **used by every client** of that interface. Two or three related methods in one interface is fine.

**"ISP is only for Java/C# with formal interfaces"**
No. In C++, pure abstract classes serve as interfaces. In duck-typed languages like Python, it's about not inheriting unused behavior from base classes.

**"Splitting interfaces makes the code more complex"**
Initially it seems like more files/classes, but it makes each unit simpler and the system as a whole much easier to evolve.
