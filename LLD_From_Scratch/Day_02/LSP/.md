# L — Liskov Substitution Principle (LSP)

## Definition

> **"Objects of a superclass should be replaceable with objects of a subclass without altering the correctness of the program."**
> — Barbara Liskov, 1987

More formally: if `S` is a subtype of `T`, then objects of type `T` may be replaced with objects of type `S` without breaking the program's expected behavior.

---

## The Substitutability Rule

If you have a pointer or reference to a base class, you must be able to swap in any derived class and have everything still work correctly — **no surprises, no exceptions, no broken contracts**.

```cpp
void process(Animal* a) {
    a->makeSound();   // Must work correctly for ANY Animal subclass
}
```

---

## Why It Matters

LSP is what makes **polymorphism trustworthy**. Without it:

- You end up writing `if (typeid(*ptr) == typeid(Derived))` guards everywhere
- Subclass overrides can silently break callers that rely on base-class behavior
- Inheritance becomes a liability instead of an asset
- Testing base-class behavior doesn't guarantee subclass behavior

---

## The Famous Square/Rectangle Problem

This is the canonical LSP violation:

```cpp
class Rectangle {
public:
    virtual void setWidth(double w)  { width = w; }
    virtual void setHeight(double h) { height = h; }
    double area() const { return width * height; }
protected:
    double width, height;
};

class Square : public Rectangle {
public:
    void setWidth(double w) override  { width = height = w; } // maintains square invariant
    void setHeight(double h) override { width = height = h; }
};
```

A `Square` **is-a** `Rectangle` geometrically, but it **violates** the behavioral contract:

```cpp
void testRectangle(Rectangle& r) {
    r.setWidth(5);
    r.setHeight(4);
    assert(r.area() == 20);  // FAILS for Square! area = 16, not 20
}
```

The `Square` subtype changed the **postconditions** of `setWidth`/`setHeight`, breaking substitutability.

---

## Formal Behavioral Contracts (Design by Contract)

LSP requires that subclasses honor these contracts:

| Contract Rule            | Meaning                                                     |
|--------------------------|-------------------------------------------------------------|
| **Preconditions**        | Subclass may only *weaken* preconditions (accept more)      |
| **Postconditions**       | Subclass may only *strengthen* postconditions (return more) |
| **Invariants**           | Class invariants must be preserved by subclass              |
| **History constraint**   | Subclass must not allow state mutations the base forbids    |
| **No exceptions surprise**| Subclass must not throw new/unexpected exception types    |

---

## Correct Design for the Rectangle Problem

Stop misusing inheritance. Use a common interface:

```cpp
struct Shape {
    virtual double area() const = 0;
};

class Rectangle : public Shape { ... };
class Square    : public Shape { ... };
```

Both share a concept (shape with area) without one pretending to *be* the other.

---

## Real-World Analogies

- A **read-only file handle** subclassing a **read-write file handle** violates LSP — callers expecting write access will fail.
- A **penguin** subclassing **bird** with `fly()` throwing an exception violates LSP — callers that call `fly()` on a `Bird*` will break.
- A **savings account** disabling `withdraw()` from a **bank account** base class violates LSP.

---

## Signs of LSP Violations

- Subclass methods throw `NotImplementedException` or `UnsupportedOperationException`
- Code checks `instanceof` / `typeid` before calling a virtual method
- Subclass overrides ignore or weaken documented postconditions
- Subclass constructors apply additional constraints not present in the base

---

## Benefits of Following LSP

| Benefit                  | Description                                              |
|--------------------------|----------------------------------------------------------|
| Reliable polymorphism    | Substitutions always behave correctly                    |
| Fewer type checks        | No need for `dynamic_cast` guards                        |
| Safer refactoring        | Swap implementations without fear                        |
| Better hierarchy design  | Forces honest "is-a" relationships                       |
