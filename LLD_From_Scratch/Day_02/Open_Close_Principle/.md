# O — Open/Closed Principle (OCP)

## Definition

> **"Software entities (classes, modules, functions) should be open for extension but closed for modification."**
> — Bertrand Meyer (1988), popularized by Robert C. Martin

You should be able to add new behavior **without changing existing, tested code**. Instead of editing a class, you *extend* it through inheritance, composition, or polymorphism.

---

## The Core Idea

| State      | Meaning                                                                 |
|------------|-------------------------------------------------------------------------|
| **Open**   | You can add new functionality (new subclasses, new implementations)     |
| **Closed** | Existing code does not need to be touched when you add that functionality|

Think of a plugin system: you add a new plugin without recompiling the host application.

---

## Why It Matters

When you modify existing code to add a feature, you risk:

- **Introducing regressions** in previously working functionality
- **Breaking existing tests** that now need updating
- **Violating the stability** of a class that other code depends on
- **Tight coupling** between features that should be independent

OCP enforces the habit: *"Don't touch what works. Extend it instead."*

---

## Classic Violation Example

```cpp
class DiscountCalculator {
public:
    double calculate(const string& customerType, double price) {
        if (customerType == "Regular")  return price * 0.95;
        if (customerType == "VIP")      return price * 0.80;
        if (customerType == "Employee") return price * 0.60;
        // Adding "Student" means editing this function — VIOLATION
        return price;
    }
};
```

Every new customer type forces you to **open and edit** `DiscountCalculator`.

---

## Correct Design

Define an abstraction; each type provides its own implementation:

```cpp
struct DiscountStrategy {
    virtual double apply(double price) const = 0;
    virtual ~DiscountStrategy() = default;
};

struct RegularDiscount  : DiscountStrategy { double apply(double p) const override { return p * 0.95; } };
struct VIPDiscount      : DiscountStrategy { double apply(double p) const override { return p * 0.80; } };
struct StudentDiscount  : DiscountStrategy { double apply(double p) const override { return p * 0.70; } };

class DiscountCalculator {
    const DiscountStrategy& strategy;
public:
    explicit DiscountCalculator(const DiscountStrategy& s) : strategy(s) {}
    double calculate(double price) const { return strategy.apply(price); }
};
```

Adding "Student" discount = adding a new class. **Zero changes to existing code.**

---

## Mechanisms for Achieving OCP in C++

| Mechanism              | How it achieves OCP                                   |
|------------------------|-------------------------------------------------------|
| Virtual functions      | Subclass overrides behavior without changing base     |
| Abstract base classes  | Define contracts; implementations are separate        |
| Templates / generics   | Compile-time polymorphism; add types without edits    |
| Strategy pattern       | Swap algorithms at runtime through a common interface |
| Visitor pattern        | Add operations to a type hierarchy without editing it |

---

## Real-World Analogies

- **USB ports**: The laptop is closed (fixed hardware). You extend it by plugging in new devices.
- **Tax system**: Tax rules change every year, but the calculator engine stays the same if designed with OCP.
- **CI/CD pipelines**: Add a new build step (plugin) without rewriting the pipeline engine.

---

## Benefits

| Benefit                    | Description                                               |
|----------------------------|-----------------------------------------------------------|
| Safer changes              | New features don't touch tested, working code             |
| Easier testing             | Each extension is independently testable                  |
| Better scalability         | The system grows without becoming a patchwork of ifs      |
| Clearer abstractions       | Forces you to identify the right extension points upfront |

---

## When to Apply OCP

Apply OCP **where variability is expected**. Not every class needs it — premature abstraction is wasteful. The right time is:

1. When you find yourself making the same kind of edit repeatedly
2. When a feature is likely to have multiple variants (payment types, report formats, discount rules)
3. When you're designing a library or framework others will extend
