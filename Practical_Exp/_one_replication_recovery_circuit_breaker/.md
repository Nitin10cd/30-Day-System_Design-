# How Each Concept Was Implemented

---

## Two Databases

Instead of a real network replica, we used two separate MongoDB databases on the same instance.

```text
primary_db   →  Accepts all writes

replica_db   →  Receives replicated data, serves reads
```

Two separate Mongoose connections created in `config/db.js`.

```text
primaryConn  →  mongodb://localhost:27017/primary_db

replicaConn  →  mongodb://localhost:27017/replica_db
```

Same schema registered on both connections separately.

This simulates how a real replica set works where both nodes have identical schema.

---

# Write-Ahead Log Implementation

WAL is stored as a MongoDB collection on the primary database.

Schema:

```text
operation          INSERT / UPDATE / DELETE

documentId         MongoDB ObjectId of the affected document

data               Full document snapshot at time of write

appliedToReplica   Boolean flag, false until replica confirms

timestamp          When the write happened
```

Flow:

```text
Write Request Arrives

        |

        v

WAL Entry Created With appliedToReplica = false

        |

        v

Write Applied To Primary DB

        |

        v

Replication Attempted

        |

 +------+------+

 |             |

Success      Failure

 |             |

WAL Entry    WAL Entry Stays

Marked       appliedToReplica = false

true         (will be recovered later)
```

This guarantees:

```text
No Write Is Ever Lost

Even If Replica Is Down At Time Of Write
```

# Replication Service

Located at `services/replicationService.js`.

Three responsibilities:

```text
replicate()   →  Log to WAL and sync to replica after every write

recover()     →  Replay all pending WAL entries when replica comes back

getLag()      →  Count how many WAL entries are still pending
```


# Circuit Breaker Implementation

Located at `services/circuitBreaker.js`.

State machine with three states:

```text
CLOSED  →  Requests pass through to replica

OPEN    →  Requests blocked, error thrown immediately

HALF_OPEN  →  One test request allowed through
```

Configuration via `.env`:

```text
CB_FAILURE_THRESHOLD = 3      Circuit opens after 3 failures

CB_RECOVERY_TIMEOUT  = 15000  Wait 15 seconds before testing recovery
```

---

## State Transitions

```text
Initial State  →  CLOSED

3 Failures     →  OPEN

15 Seconds     →  HALF_OPEN

Test Passes    →  CLOSED

Test Fails     →  OPEN Again
```

---

## What Happens In Each State

```text
CLOSED

Request → Circuit → Replica → Response

OPEN

Request → Circuit → Error Thrown → Fallback To Primary

HALF_OPEN

Request → Circuit → One Test Request → Replica

                         |

                    Pass → CLOSED

                    Fail → OPEN
```

# Data Recovery Implementation

Located inside `replicationService.js` as the `recover()` function.

Called manually via:

```text
POST /api/recover
```

Flow:

```text
Recovery Triggered

        |

        v

Find All WAL Entries Where appliedToReplica = false

Sort By Timestamp Ascending

        |

        v

For Each Pending Entry

        |

        v

Call Circuit Breaker

        |

 +------+------+

 |             |

Circuit       Circuit

Closed        Open

 |             |

Apply Entry   Stop Recovery

To Replica    Return Error

 |

Mark Entry

appliedToReplica = true

        |

        v

Next Entry

        |

        v

All Entries Applied

Replica Fully In Sync
```

# Complete Request Flow

## Normal Write Flow

```text
POST /api/products

      |

Product Created On primary_db

      |

WAL Entry Logged  (appliedToReplica: false)

      |

Circuit Breaker → CLOSED

      |

Product Written To replica_db

      |

WAL Entry Updated  (appliedToReplica: true)

      |

Response Returned To Client

source: "primary"
```

---

## Replica Down Flow

```text
POST /api/products

      |

Product Created On primary_db

      |

WAL Entry Logged  (appliedToReplica: false)

      |

Circuit Breaker → Replica Unreachable

      |

Failure Count Incremented

      |

After 3 Failures → Circuit OPENS

      |

WAL Entry Stays Pending

      |

Response Still Returned To Client

Data Is Safe In Primary + WAL
```

---

## Recovery Flow

```text
Replica Comes Back Online

      |

.env Fixed With Correct URI

      |

Server Restarted

      |

POST /api/recover Called

      |

WAL Queried For Pending Entries

      |

Each Entry Replayed On Replica In Order

      |

All Entries Marked appliedToReplica: true

      |

GET /api/lag Returns inSync: true

      |

Replica Fully Caught Up
```

# What We Observed

```text
Observation 1

Every write was logged to WAL before replication was attempted.
No data was lost even when replica was completely unreachable.

Observation 2

Circuit breaker opened after exactly 3 failures.
Subsequent requests were blocked immediately without waiting for timeout.
This saved significant time compared to waiting for each request to fail.

Observation 3

When circuit was OPEN, reads automatically fell back to primary.
Users would have seen no downtime from the client side.

Observation 4

After restoring replica connection and triggering recovery,
all pending WAL entries were replayed in chronological order.
replica_db came back to full sync with primary_db.

Observation 5

GET /api/lag confirmed inSync: true after recovery.
MongoDB Compass showed identical data in both databases.
```

---

# Key Learnings

```text
WAL is the single source of truth for recovery.
If WAL is intact, no data is ever permanently lost.

Circuit breaker protects the entire system.
Without it, a slow replica would slow down every single request.

Fallback to primary ensures zero downtime for reads.
Users never see an error just because replica is down.

Recovery must replay WAL in order.
Out of order replay causes data inconsistency.

This same pattern is used by PostgreSQL WAL,
MySQL binlog, and MongoDB oplog in production systems.
```

---

# Summary

- We built a Product API with full replication between two MongoDB databases.
- Every write is first logged to WAL, then replicated to replica.
- If replica is down, WAL stores the entry with appliedToReplica set to false.
- Circuit breaker opens after 3 consecutive replica failures.
- When circuit is open, reads fall back to primary automatically.
- After 15 seconds circuit moves to HALF_OPEN and tests recovery.
- Recovery endpoint replays all pending WAL entries in chronological order.
- After recovery, replica is fully in sync with primary.
- This mirrors exactly how PostgreSQL streaming replication and WAL recovery works.
- The entire flow was verified end to end using Postman and MongoDB Compass.
